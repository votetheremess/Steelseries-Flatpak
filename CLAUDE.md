# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Linux ChatMix + battery monitoring app for the **SteelSeries Arctis Nova Elite** (USB `1038:2244`). SteelSeries provides no Linux software for this headset; the GameDAC's ChatMix feature is Windows-only via SteelSeries GG. This project reverse-engineered the HID protocol and reimplements ChatMix by creating virtual PipeWire sinks and reading dial events from the device, with a Rust + GTK4/libadwaita GUI and a KDE system tray icon. Phases 1–5 (CLI, GUI, persistence, background service mode, clip-system) all landed on main as of 2026-05-14. Flatpak packaging (eventual Phase 6) is not yet started.

## Build & run

The dev environment lives in a distrobox named `fedora-dev` because Bazzite (the user's host) is immutable Fedora and doesn't have GTK4/libadwaita dev packages on the host. Rust is installed on the host via rustup (in `~/.cargo`, which is shared into the distrobox via `$HOME`).

**Build inside distrobox, run on the host.** File edits happen on the host (shared `$HOME`), cargo dispatches into the distrobox. The *binary* itself should run on the host whenever possible — that's the real target environment (Bazzite, with the user's actual PipeWire/HID/display), and running there avoids any distrobox-specific quirks (socket paths, device perms, theming differences). Running inside the distrobox works too, but it's only useful as a fallback if something's broken on the host.

```bash
# Build / test — ALWAYS dispatched into the distrobox
distrobox enter fedora-dev -- cargo build
distrobox enter fedora-dev -- cargo test
distrobox enter fedora-dev -- cargo test parse_dial_position     # single test

# Quick type-check (faster than full build)
distrobox enter fedora-dev -- cargo check

# Lint / format (not enforced — clippy has known warnings, fmt has diffs)
distrobox enter fedora-dev -- cargo clippy
distrobox enter fedora-dev -- cargo fmt --check

# Run — PREFER the host (Bazzite has GTK4 runtime libs)
./target/debug/arctis-chatmix
./target/debug/arctis-chatmix --hidden     # launch hidden (for autostart)

# Fallback only: run inside distrobox if the host run fails
distrobox enter fedora-dev -- ./target/debug/arctis-chatmix
```

Never run `cargo build` on the host — it will fail because GTK4/libadwaita/dbus dev headers aren't there.

Rust edition 2024. Key feature gates: `gtk4` v4_12, `libadwaita` v1_5, `ksni` 0.2. HTTP client is `ureq` (default features off, `tls` for rustls) — used only for fetching Sonar share-URL presets. `serde_json` + `base64` are pulled in specifically for parsing Sonar's JSON shape and deeplink payloads (see "Sonar preset compatibility" below); the earlier "no serde_json" rationale in this doc applies only to hand-parsing `pactl` line-oriented output, not to arbitrary web JSON. Tests are colocated in source files — 9 HID protocol parsing tests in `src/hid/protocol.rs`, 10 biquad EQ math tests in `src/eq/biquad.rs` (Peaking / Low Shelf / High Shelf / Low Pass / High Pass / Notch + disabled + combined + log-frequency helpers), 5 volume-mode mixer tests in `src/audio/mixer.rs` (dead code), 9 Sonar import tests in `src/eq/sonar_import.rs` (covers flat / `data.*` / `data.data.*` CDN shapes, share-URL decoding, deeplink host validation, value clamping, unknown-type errors), 7 spatial state tests in `src/eq/spatial.rs` (default GSX profile, profile round-trip, unknown-profile fallback, legacy-format parsing including SOFA-era noise, wet_mix clamping, garbage tolerance, profile-key uniqueness), 6 reconcile_tests + 4 mic_preference_tests in `src/audio/persistence.rs`, 7 hotplug_tests in `src/audio/router.rs`, 1 parse_device_list test in `src/audio/sinks.rs`, 1 smoke test in `src/audio/state_sync.rs`, 18 `BufferController` state-machine tests in `src/clips/buffer.rs` (pause/resume, onboarding-gate arming, save-during-pause, portal reset, config-change deferral, etc.), 26 backend tests in `src/clips/backend.rs` (FIFO validation, restart-storm guard, command dispatch), 9 clip-browser tests in `src/clips/browser.rs`, 8 remix tests in `src/clips/remix.rs`, 6 retention tests in `src/clips/library.rs`, 3 settings round-trip tests in `src/clips/settings.rs`, plus a smaller scatter across `clips/mod.rs`, `clips/gsr_install.rs`, `clips/portal.rs`, and `desktop_file.rs`. **136 tests total** (1 additional ignored; 137 `#[test]` attributes overall).

The repo bundles three HeSuVi HRIR `.wav` files in `data/hrtf/` (gsx.wav, dh2.wav, dts-hpx.wav — total ~730 KB) used by the spatial audio feature. They're embedded into the binary via `include_bytes!` and extracted to `~/.cache/arctis-chatmix/hrtf/` lazily per profile when the user enables spatial. Attribution is in `LICENSES/hesuvi.txt`.

## Architecture — module layout

The app is structured as a persistent background process with a libadwaita GUI and a system tray icon. Four threads communicate via `std::sync::mpsc` channels polled by glib timers:

- **GTK main thread** — glib event loop, all UI updates + audio commands
- **HID listener thread** — blocking `libc::poll` reads, sends `HidEvent`s (polled every 50ms)
- **Tray thread** — `ksni` service, sends `TrayCommand`s (polled every 100ms)
- **EQ pipeline thread** — persistent `pw-cli` child process, receives `EqCommand`s from the GTK main thread, manages PipeWire filter-chain modules for per-sink EQ

- **`src/main.rs`** — thin entry point. Parses `--hidden` flag, calls `app::run(hidden)`.
- **`src/app.rs`** — owns the `adw::Application` lifecycle. `connect_startup` runs `init_pipeline` (sinks, sources, router, HID device, ChatMix enable command). `connect_activate` builds the window on first activation, reactivates the existing hidden window on subsequent invocations. Uses `app.run_with_args(&[])` to prevent GTK from parsing `--hidden` (which it rejects as unknown). Builds four sets of callbacks for the window: EQ apply (16ms debounce → `router.update_eq`), spatial apply (16ms debounce → `router.set_spatial`), mixer reroute (`router.reroute_sink`), and mic reroute (`router.reroute_mic`). `AppResources` stores `headset_sink: String` so the mixer can control the physical headset volume. Wires four glib timers + per-callback debounces:
  1. HID event poll (50ms) — drains the mpsc from the listener thread, updates widgets + sink volumes
  2. Battery query (30s) — sends `0x06 0xB7` via the HidWriter
  3. **State-sync tick (2s)** — calls `state_sync::tick` which orchestrates: stream-routing reconciliation via `persistence::reconcile_stream_state` (renamed from `restore_new_streams`); mic hotplug detection via `AudioRouter::check_mic_hotplug`; virtual-source volume capture via `state_sync::capture_virtual_volumes`.
  4. Tray command poll (100ms) — receives Show/Quit commands from the `ksni` tray thread
  5. EQ debounce (16ms one-shot) — when the EQ UI changes, a 16ms `timeout_add_local_once` batches rapid drag events and sends the band settings to the EQ pipeline thread via `router.update_eq()`. The spatial slider drag uses the same pattern via `router.set_spatial()`.
- **`src/window.rs`** — horizontal dashboard layout (1200x725). Top-level structure: sidebar rail (70px, always visible) + content area (HeaderBar + `gtk::Stack` with Crossfade transitions). Stack pages: `"home"` (dashboard), `"eq"` (parametric equalizer + mixer — see `src/eq/` and `src/mixer.rs`), `"clips"` (onboarding wizard + library browser, see `src/clips/`), `"engine"` (StatusPage placeholder, insensitive sidebar button), `"settings"` (PreferencesPage with autostart/clear/quit). Dashboard page has a 2-column grid for row 0 (Status card with battery + ChatMix slider, and Device card with connection + noise control), then a **full-width row 1 "Clips" card** with a single horizontal row: `[Save Clip] [● Capturing | Start Capturing] <spacer> [Duration: 60s] [Quick Capture: Alt + S]`. The capture-toggle button carries a colored pulsing dot (`.dot-capturing` green when armed, `.dot-paused` muted gray when paused/idle, `.dot-saving` while saving, `.dot-error` red on error) and the verb flips between "Capturing" (live) and "Start Capturing" (paused/idle). The Save button fires `app.save-clip`; the capture-toggle button fires `app.pause-recording-toggle` (action name kept stable for any D-Bus scripts the user has wired up even though the UI verb changed); Duration and Quick Capture are flat buttons that fire the new `app.show-clips-settings` GAction to jump straight to the Settings sidebar tab via `ChatMixWindow::show_settings_tab()` (mirror of `show_clips_tab()`). `gtk::SizeGroup` enforces matched row/card heights on row 0. Sidebar uses `gtk::ToggleButton`s grouped via `set_group()`. CSS provider scales text 125% and adds row padding + EQ/mixer/spatial-specific classes (`.eq-floating-panel`, `.mixer-channel`, `.spatial-mix-pill`, etc.), plus `.clip-dot` and per-state `.dot-paused / .dot-capturing / .dot-saving / .dot-error` classes for the Clips card indicator. `ChatMixWindow::new` accepts EQ apply, **spatial apply**, reroute, mic reroute callbacks + headset sink name. `Widgets` struct holds `mixer: Option<MixerWidgets>` for HID-driven volume updates and `clips_section: Option<ClipsSectionWidgets>` for the home-page clips card. `ClipsSectionWidgets` exposes `refresh(state, user_paused, settings)` (full refresh including hotkey/duration labels) and `refresh_state(state, user_paused)` (dot + toggle button only — used for high-frequency state transitions). `set_clips_state(state, user_paused)` is the public window-level entry point; both args are required because the `Saving + user_paused = true` window means state alone can't be used to derive `user_paused` without flicker. Public API: `set_chatmix`, `set_battery`, `set_noise_mode`, `set_connected`, `set_sink_volume`, `set_clips_state`, `refresh_clips_section`, `show_clips_tab`, `show_settings_tab`.
- **`src/tray.rs`** — `ksni`-based StatusNotifierItem (SNI) tray icon. Spawns on its own thread via `TrayService::spawn()`. Communicates with GTK main thread via `mpsc::channel<TrayCommand>`. Left-click = Show window, right-click menu = Show Window / Quit. Uses system `audio-headphones` icon (not Lucide — tray hosts query their own theme).
- **`src/hid/`** — HID protocol layer. `device.rs` finds/opens hidraw via `/sys/class/hidraw/*/device/uevent`. `protocol.rs` parses both report-ID `0x07` events and `0x01` responses. `listener.rs` spawns a thread doing blocking reads via `libc::poll`.
- **`src/mixer.rs`** — mixer view embedded in the EQ page. `build_mixer_content()` returns a horizontal row of 6 card-styled channels (Master, Game, Chat, Music, Aux, Mic). Each strip is a fixed 150px-wide `gtk::Box` with CSS classes `.mixer-channel` (card: bg + border + radius matching `.eq-floating-panel`) plus a per-channel accent class `.mixer-ch-{master,game,chat,music,aux,mic}` that sets the top-border `box-shadow`, icon color, and slider trough highlight. Accent colors are drawn from `BAND_COLORS` in `src/eq/model.rs`. Master has no device dropdown; its dropdown slot is an empty `gtk::Box` joined to a shared `gtk::SizeGroup` (Vertical mode) with the other dropdown wrappers, so all strips have identical height. Volume changes call `sinks::set_sink_volume`/`set_source_volume` directly. Device changes fire `on_reroute`/`on_mic_reroute` callbacks wired to the router. Returns `MixerWidgets` with scale refs for external updates (ChatMix HID events sync Game/Chat sliders via `ChatMixWindow::set_sink_volume`). Uses `Rc<Cell<bool>>` `updating` flag to suppress `value_changed` during programmatic updates. Device dropdowns use two custom `gtk::SignalListItemFactory` instances: `truncating_label_factory` (button: `max_width_chars(1)` + `EllipsizeMode::End` + tooltip with full name) and `popup_label_factory` (popup list: word-wrapped full names). The mic dropdown also attaches a `gtk::GestureClick` in Capture phase that re-queries `sinks::list_physical_sources()` and rebuilds the `StringList` model before the popup opens, so mics that come online after app startup appear. Selection is preserved across refresh by matching the internal device name.
- **`src/audio/`** — shells out to `pw-cli`, `pw-link`, and `pactl`.
  - `sinks.rs` — creates 4 virtual output sinks via `pw-cli create-node adapter` with `media.class=Audio/Sink` (Game, Chat, Music, Aux — stereo `FL,FR`) and 1 virtual mic source with `media.class=Audio/Source/Virtual` (mono `MONO`). Sink names: `SteelSeries_Game`, `SteelSeries_Chat`, `SteelSeries_Music`, `SteelSeries_Aux`. Mic source: `SteelSeries_Mic`. The mic is `Audio/Source/Virtual` (not a sink) so it appears as a selectable input device in Discord, OBS, etc. Contains `LEGACY_SINK_MIGRATIONS` for migrating old `ChatMix_*` names. Also provides device enumeration (`list_physical_sinks`, `list_physical_sources` — parses `pactl list sinks`/`sources` verbose output, filters out virtual/EQ nodes; both now return 5-tuples `(name, desc, Option<product_name>, Option<bus_path>, Option<bluez_address>)` so the mic hotplug logic can match preferred devices across name churn) and volume queries (`get_sink_volume`, `get_source_volume`, `set_source_volume`). A `parse_device_list` unit test exercises the parser against representative `pactl` output.
  - `router.rs` — owns the **EQ pipeline**, **spatial pipeline**, mic linking, and **per-sink device routing**. For output sinks: routes audio through PipeWire `libpipewire-module-filter-chain` modules loaded via a persistent `pw-cli` child process on a dedicated thread (`eq-pipeline`). Two graph topologies depending on `SpatialState.enabled`:
     - **Simple chain** (spatial off): 10 biquad nodes `eq0..eq9`, PipeWire auto-replicates per channel.
     - **Spatial chain** (spatial on): explicit dual EQ (`eq_l_0..eq_l_9` + `eq_r_0..eq_r_9`) → 7 stereo→7.1 upmix mixers → 14 HeSuVi convolver nodes → 2 output mixers (sum 7 wet HRIR outs + 1 dry bypass per side). Single module per sink, ~43 nodes when on.
     The EQ thread receives `EqCommand::Apply`, `EqCommand::Reroute`, `EqCommand::SetSpatial`, and `EqCommand::Shutdown`. `Apply` updates EQ band controls in-place via `set-param` (zero audio gap) — uses `eq_l_*`/`eq_r_*` prefix when spatial is active, else `eq*`. Filter-type changes still trigger a full rebuild. `SetSpatial` rebuilds when topology changes (enable toggle or profile change), but **wet_mix changes go through `update_wet_mix()` which set-params the `mix_out_l/r:Gain N` controls live (no rebuild, no audio gap).** `Reroute` destroys and rebuilds a filter-chain with a new `target.object` to change which physical device a sink routes to. Per-sink targets + per-sink spatial state are tracked in `HashMap`s initialized from saved state on startup. For mic: `reroute_mic()` unlinks the current source (parsing `pw-link -l` to find ALL connections to `SteelSeries_Mic:input_MONO`) and links the new one. **Mic hotplug fields**: `current_mic_source: Option<String>` (last-known live link target) and `preferred_mic: Option<MicPreference>` (user's saved preference). `AudioRouter::check_mic_hotplug` is the periodic-tick entry point: it re-enumerates `list_physical_sources`, calls the pure helper `decide_hotplug_target` (`pub(crate)` for unit tests) with the current source list + current source + preferred mic, and if the helper returns a new target, calls `reroute_mic`. `decide_hotplug_target` uses a 4-tier match priority: exact node name → `api.bluez5.address` → `device.bus_path` → `device.product.name`. On startup, sets `SteelSeries_Game` as default sink and `SteelSeries_Mic` as default source via `pactl set-default-sink/source` so WirePlumber routes new streams through the virtual pipeline. Includes `cleanup_orphaned_loopbacks()` for legacy cleanup.
  - `persistence.rs` — saves/restores app→sink assignments to `~/.config/arctis-chatmix/assignments.txt`. `reconcile_stream_state()` (renamed from `restore_new_streams` as of 2026-05-14) is polled every 2s by `state_sync::tick` to auto-route newly launched apps AND distinguish app-initiated moves from user-initiated moves; the underlying pure planner `reconcile_pure_with_updates` uses a 3-second timestamped `SUPPRESSION_WINDOW` to debounce pactl eventual-consistency lag immediately after the app issues a move. Skips sink-inputs with `app_name == "pw-cli"` to avoid moving EQ filter-chain streams (which would create feedback loops). `update_saved_assignment(app, sink)` is the monotonic single-entry writer used by reconcile: it only ever **adds or updates** an entry mapping `app` onto a managed sink, and is paired with the rule that `save_assignments` never destroys entries when streams move to unmanaged sinks (transient routing churn cannot wipe the user's saved mappings). `initial_tracked()` seeds the state-sync `tracked: HashMap<u32, String>` map at startup by reading current sink-input → sink pairings, so the first tick after launch doesn't treat all existing streams as "new". `clear_saved()` deletes the config file (used by the Clear Config button). Also handles mixer device routing persistence (`load_mixer_routing`, `save_mixer_routing_entry`) in `~/.config/arctis-chatmix/mixer_routing.txt` — `load_mixer_routing` **skips** the `mic` line by design; use `load_mixer_routing_mic` for that record. The mic line is **5 tab-separated fields**: `mic\t<node>\t<product>\t<bus_path>\t<bluez_addr>` (backward-compatible with the legacy 2-field and 3-field shapes). `save_mixer_routing_mic(MicPreference)` writes the full 5-field form. **`MicPreference`** is a struct with `node_name`, `product_name`, `bus_path`, `bluez_address` (the latter three `Option<String>`) used by `decide_hotplug_target` in router.rs to match a preferred device across PipeWire-node name churn. `parse_mic_preference_from` and `parse_mixer_routing_from` are the pure parser entry points used by both the runtime loader and the `mic_preference_tests` / `reconcile_tests` suites. Also handles **mixer volume persistence** (`load_volumes`, `save_volume_entry`) in `~/.config/arctis-chatmix/volumes.txt`. Volumes are saved only for Music / Aux / Mic — Game and Chat are HID-driven (the ChatMix dial overrides any saved value), and Master is the physical headset that WirePlumber persists natively.
  - `state_sync.rs` — unified state-sync tick orchestrating Issues 1+2+3 from the 2026-05-13 routing/volume/clips spec. `StateSyncState` holds `tracked: HashMap<u32, String>` (sink-input-id → last-known sink), `self_move_pre_sink: HashMap<u32, (String, Instant)>` (post-move suppression window, 3s, keyed by the sink the stream was on BEFORE we moved it), and `last_saved_volumes: HashMap<String, u32>` (debounces `volumes.txt` writes against unchanged values). `tick(&mut state, &mut router)` runs once per 2s on the GTK main thread: calls `persistence::reconcile_stream_state` (assignments), then `router.check_mic_hotplug` (mic), then `capture_virtual_volumes` (volumes). No new threads, no internal locking — single owner on the GTK main thread. Includes a smoke test verifying that one tick on an empty/initial state is a no-op.
  - `mixer.rs` — currently unused (dead code kept for Volume-mode fallback reference).
- **`src/eq/`** — 10-band parametric equalizer with live PipeWire filter-chain backend, plus integrated mixer view and per-sink HeSuVi-HRIR spatial audio. Per-sink EQ for Game, Chat, Music, Aux; spatial only on Game/Music/Aux (Chat and Mic skip both UI and DSP, guarded by `EqTarget::supports_spatial()`). State shared via `Rc<RefCell<EqState>>` (same pattern as `ChatMixWindow`). `build_eq_page()` accepts EQ apply, **spatial apply**, reroute, mic reroute callbacks + headset sink name, returns `(gtk::Widget, Option<MixerWidgets>)`. The EQ page top bar has a `[Mixer]` toggle button (separate pill, not visually linked) to the left of the `[Game|Chat|Music|Aux|Mic]` linked tab group — all in the same radio group for mutual exclusivity. **The Mixer tab is activated by default at the end of `build_eq_page` (after all the stack/refs are wired up) so the EQ page lands on the volume mixer the first time the user enters it.** On the right: `[Show All] [Preset dropdown] [✏️ Edit] [Import]`. The Edit icon button opens `show_edit_dialog` — a popup with an editable name field (disabled for built-ins), **Cancel** (discards unsaved name change), **Reset** (reverts bands; commits any pending rename first), **Delete** (removes the file, falls back to Flat; disabled for built-ins), and **Save** (suggested-action; commits the rename — Enter in the name entry triggers Save too). The Import button opens `show_import_dialog` — a two-phase flow: paste URL → click Fetch → URL goes read-only + name field pre-fills with Sonar's name (selected so typing replaces) → Save writes a custom preset with the (possibly edited) name. Below the band-editor pill sits the **spatial panel** (visible only on Game/Music/Aux): a horizontal pill with the title, profile dropdown, a `.spatial-mix-pill`-styled inline Dry/Wet slider (180px wide, fixed height matching the dropdown), and the enable switch on the right. Clicking Mixer switches a content `gtk::Stack` from `"eq"` (graph + controls + spatial) to `"mixer"` (volume bars + device dropdowns from `src/mixer.rs`) and hides the EQ-only controls + spatial panel. Clicking any sink tab switches back.
  - `model.rs` — data model: `Band` (freq/gain/Q/filter type), `FilterType` (Peaking, LowShelf, HighShelf, LowPass, HighPass, **Notch** — added for Sonar parity), `SinkEq` (bands + preset_name + **spatial**), `EqTarget` (with `supports_spatial()` returning false for Chat/Mic), `EqState` with `Option<usize>` selected band. 10 band colors, default frequencies at 31–16kHz. **`SpatialProfile`** enum (Gsx, Dh2, DtsHpx) with `key()`/`from_key()` for stable persistence and `display_label()` for the dropdown. **`SpatialState { enabled, profile, wet_mix }`** with `WET_MIN/MAX/DEFAULT` constants (default wet 0.7). `SinkEq.preset_name` is NOT cleared when the user drags bands around — the dropdown keeps showing the active preset even after edits, so Reset knows what the user's "original" to revert to. Clearing it would make Reset lose the anchor.
  - `biquad.rs` — Audio EQ Cookbook math: computes frequency response magnitude (dB) for drawing curves. Pure math, no audio processing. `magnitude_db()`, `combined_magnitude_db()`, `log_frequencies()`. 10 unit tests (one per filter type + combined/disabled/log-freq helpers).
  - `graph.rs` — Cairo `DrawingArea` with log-frequency X axis (20Hz–20kHz) and linear dB Y axis (±12dB). Draws grid, individual band curves (colored fills + lines), combined response curve (white), and numbered control points. Single `GestureDrag` handles both click-to-select/deselect and drag-to-move (tracks `did_drag_move` to distinguish). `EventControllerScroll` for Q adjustment. "Show All" toggle controls whether all bands or only the active band's curve is visible.
  - `controls.rs` — Blender-style scrub entries: `GestureDrag` on `gtk::Entry` in `Capture` phase intercepts before GTK's text handling. Drag left/right to adjust value (freq=log, gain=linear, Q=log). Click without drag enters text edit mode. `EventControllerMotion` continuously forces `ew-resize` cursor when not editable. Labeled pills layout: `Freq [125 Hz]` `Gain [+3.0 dB]` `Q [1.00]` `Type [Peaking ▾]` `On [toggle]`. GTK's built-in `DragSource` is removed from each Entry at creation to prevent crashes (`gtk_text_util_create_drag_icon: assertion 'text != NULL'` when GTK's text DnD conflicts with our scrub gesture).
  - `presets.rs` (pub) — 7 built-in presets (Flat, Bass Boost, Treble Boost, V-Shape, FPS Competitive, Vocal Clarity, Warm). Custom preset save/load to `~/.config/arctis-chatmix/eq_presets/`. Per-sink EQ state persisted to `~/.config/arctis-chatmix/eq_state.txt` as **4 tab-separated fields**: `<sink_name>\t<preset_or_custom>\t<inline_bands>\t<spatial_blob>`. The 4th field is optional — old 3-field state files (and even older 2-field ones) parse cleanly with `SpatialState::default()` for the missing data. Public because `router.rs` calls `load_eq_state()` at startup to initialize filter-chains with saved settings.
  - `spatial.rs` (pub) — bundled HeSuVi HRIR extraction + spatial state serialization. `embedded_ir_bytes(profile)` returns the `include_bytes!`'d slice for the chosen profile; `ensure_ir_file(profile)` writes that slice to `~/.cache/arctis-chatmix/hrtf/<key>.wav` lazily (size-mismatch check is the cheap idempotence — full content compare would re-read 0.5–1 MB every launch). `serialize`/`parse` handle the `enabled=1;profile=gsx;wet=0.700` blob, with backward-compat for missing fields and silently-ignored legacy keys (`pi=…;d=…;fa=…` from the SOFA-era prototype). 7 unit tests cover defaults, round-trip per profile, unknown-profile fallback, legacy-blob parsing, wet_mix clamping, garbage tolerance, and profile-key uniqueness.
  - `sonar_import.rs` (pub) — fetches a SteelSeries Sonar preset from a share URL or raw CDN URL and maps it into our `Band` model. Public entry points: `import_from_url` (deeplink or CDN URL → HTTP fetch → `ImportedPreset`) and `parse_sonar_json` (string → `ImportedPreset`, used in tests and for files produced by `tools/dump-sonar-presets.ps1`). Share URLs look like `https://www.steelseries.com/deeplink/gg/sonar/config/v1/import?url=<base64>`; the base64 decodes to a `https://community-configs.steelseriescdn.com/v1/<uuid>` URL that returns the JSON anonymously. Decoded deeplinks are validated to point at the CDN host before we fetch, so a malicious share URL can't redirect us to an arbitrary server. **The JSON parser walks `serde_json::Value` generically** instead of using strongly-typed structs — Sonar ships at least three wrapping shapes: flat (`{"name": ..., "parametricEQ": ...}` — DB row form), single-wrapped (`{"name": ..., "data": {"parametricEQ": ...}}`), and double-wrapped (`{"metadata": {"name": ...}, "data": {"data": {"parametricEQ": ...}}}` — the actual CDN share-link form, confirmed against `community-configs.steelseriescdn.com` responses). `find_parametric_eq` DFS-walks for the first object under a `"parametricEQ"` key; `find_preset_name` prefers `metadata.name` (what the sharer typed) over top-level `name`. Filter mapping: `peakingEQ|lowShelving|highShelving|lowPass|highPass|notchFilter` → `FilterType::{Peaking, LowShelf, HighShelf, LowPass, HighPass, Notch}`. Unknown types error rather than silently fall back, since a new Sonar filter type would silently miscompute. Values clamped to `FREQ_MIN/MAX`, `GAIN_MIN/MAX`, `Q_MIN/MAX` defensively. The UI (`src/eq/mod.rs` → `show_import_dialog`) runs the HTTP fetch on a spawned thread and polls via `glib::timeout_add_local` to stay off the GTK main thread; once fetched, the dialog flips into a second phase where the user can edit the preset name before saving.
- **`src/clips/`** — instant-replay-style clip recording, gated behind a portal-based screen-capture session. Modules: `mod.rs`, `backend.rs`, `buffer.rs`, `browser.rs`, `gsr_install.rs`, `hotkey.rs`, `library.rs`, `notifications.rs`, `portal.rs`, `remix.rs`, `settings.rs`, `thumbnail.rs`. ## HID protocol — reverse-engineered bytes

All events/responses on the hidraw matching interface 3 (64-byte packets). There are **two report IDs**:

**`0x07` = async events** (device → host, spontaneous on user interaction):

| Feature | Meaning | Bytes |
|---|---|---|
| `0x25` | Dial position (Volume mode only, 0–15) | byte2 = position |
| `0x45` | ChatMix levels (after enable command) | byte2 = game 0–100, byte3 = chat 0–100 |
| `0xBD` | Noise control | byte2: `0x00`=Off, `0x01`=Transparency, `0x02`=ANC |
| `0xB8` | ANC hardware event | `0x03` observed |

**`0x01` = query responses** (device → host, reply to a `0x06` write):

| Feature | Meaning | Bytes |
|---|---|---|
| `0xB7` | Battery status | byte2 = headset 0–100, byte3 = spare 0–100, byte4 = flags |

**Host → device writes** (report ID `0x06`):

- `06 49 01` — enable ChatMix mode. **Critical**: without this, ChatMix mode on the GameDAC sends nothing; only Volume mode produces `0x25` events. App sends disable→enable on startup to force a clean state reset.
- `06 49 00` — disable ChatMix mode
- `06 B7` — query battery status (sync — reply comes on report ID `0x01`)

**Dial click sends nothing** — the mode switch (Volume ↔ ChatMix) is entirely internal to the GameDAC. Software cannot tell which mode the dial is in.

## Critical gotchas

### Audio pipeline

- **`pw-cli` null sinks require `monitor.channel-volumes=true` and `monitor.passthrough=true`** in their props, or `pactl set-sink-volume` will be silently ignored (volume reports as changed, audio is unaffected). `pactl load-module module-null-sink` sets these automatically but truncates descriptions with spaces, so we use `pw-cli create-node adapter` with these properties explicitly set.
- **Sink names use underscores** (`SteelSeries_Game` not `SteelSeries Game`). Display names come from `node.description` set via `pw-cli` which supports spaces.
- **Orphaned loopback cleanup is critical**. `AudioRouter::create` calls `cleanup_orphaned_loopbacks()` which scans `pactl list modules short` for any `module-loopback` referencing our sink/source names (current AND legacy) and unloads them BEFORE creating new ones. Without this, multiple loopbacks accumulate across app restarts and produce a staticky hum/buzz on the headset output.
- **`timeout` kills without running Drop handlers** — orphaned sinks/loopbacks accumulate. `VirtualSinks::create()` calls `cleanup_orphaned()` on startup to recover.
- **Mic is `Audio/Source/Virtual`, not a sink** — created as a proper PipeWire source so it appears as a selectable input device in Discord, OBS, etc. (sink `.monitor` sources are hidden from app input device lists). The headset mic is connected via `pw-link` (direct port link), not `module-loopback`. The mic is separate from the EQ pipeline — only the 4 output sinks get filter-chains.
- **EQ filter-chains use a persistent `pw-cli` session** — modules loaded via `pw-cli load-module libpipewire-module-filter-chain` only persist while the pw-cli client is connected. A dedicated `eq-pipeline` thread keeps the session alive. The pw-cli stdout is continuously drained by a `pw-cli-reader` thread to prevent pipe buffer deadlock. On app shutdown, killing pw-cli automatically cleans up all filter-chain modules.
- **EQ live updates use `set-param` on the capture node** — `set-param <node-global-id> Props { params = [ "eq0:Freq" 1000.0 "eq0:Q" 1.5 "eq0:Gain" 3.0 ] }`. The target must be the capture NODE's global ID (not the module proxy ID). The format is a flat `params = [ key value ... ]` array, not SPA key=value pairs. Changing filter types (e.g. Peaking→LowShelf) requires a full destroy+reload since the biquad label is set at creation.
- **EQ filter-chain nodes need `node.dont-reconnect = true`** — without this, WirePlumber can re-route the playback stream to a virtual sink on headset disconnect, creating a feedback loop (sink → filter → sink → ...). Also set `node.passive = true` and `stream.dont-remix = true`.
- **`reconcile_stream_state` must skip `pw-cli` streams** — the persistence module's stream reconciliation (driven from `state_sync::tick` every 2s) skips sink-inputs with `app_name == "pw-cli"` to prevent moving EQ filter-chain streams, which would create feedback loops. (Renamed from `restore_new_streams` as of 2026-05-14 — the function does more than restore now: it also distinguishes app-initiated from user-initiated moves and propagates user moves to the save file.)
- **EQ pipeline must skip the mic** — the EQ thread's `Apply` handler checks `sink_name == MIC_SOURCE_NAME` and skips. Creating a filter-chain for `SteelSeries_Mic` routes mic audio to the headset speaker and causes WirePlumber to rewire mic source connections (swapping in the wrong physical mic). The mic has no EQ — audio goes through `pw-link` only.
- **`unlink_mic_source` must disconnect ALL links, not just the headset** — parses `pw-link -l` to find every source connected to `SteelSeries_Mic:input_MONO` and disconnects each one. The old approach of only unlinking `find_headset_source()` failed when the user had rerouted to a different mic.
- **`link_mic_source` must pick an explicit output port** — `pw-link <src_node> SteelSeries_Mic` silently drops the link when channel counts don't match (e.g. a stereo mic with `capture_FL`/`capture_FR` vs. our mono `input_MONO`): pw-link pairs FL→MONO, then silently fails on FR because there's no matching input. The fix enumerates the source's output ports via `pw-link -o` and picks one explicitly — prefers `:capture_MONO`, falls back to `:capture_FL`, then the first port. Bare-node linking is kept as a last-resort fallback.
- **`reroute_mic` persists intent BEFORE linking** — `save_mixer_routing_entry("mic", new_source)` runs first; the `pw-link` attempt runs after. If the live link fails (port mismatch, device disappeared), the user's selection is still remembered so the dropdown reflects their intent and the next launch tries again. The old order (save only on success) meant a single failed link silently reverted the saved routing to whatever was there before, confusing the user into thinking "it doesn't remember my selection."
- **Default sink/source must be set on startup** — `pactl set-default-sink SteelSeries_Game` and `pactl set-default-source SteelSeries_Mic` ensure WirePlumber routes new streams through the virtual pipeline. Without this, apps like Discord connect directly to physical devices, bypassing the virtual sinks entirely, and the mixer device dropdowns have no effect.
- **Per-sink output targets require filter-chain rebuild** — `target.object` is set at module load time and cannot be changed via `set-param`. Rerouting a sink to a different physical device destroys the old filter-chain and loads a new one (brief ~300ms audio interruption). Targets are tracked per-sink in `HashMap<String, String>` in the EQ thread.
- **Mixer dropdowns pre-select the currently-routed device on build** — both `build_sink_dropdown` and `build_source_dropdown` load `mixer_routing.txt` and (for the mic) fall back to `find_headset_source()`, mirroring the init logic in `AudioRouter::create`. Without this the dropdowns default to index 0 of the device list, which silently misrepresents what the backend is actually doing — users see "Alias Pro" when the mic is actually linked to the Arctis. The pre-select runs before the `connect_selected_notify` handler is wired, so it doesn't cause a spurious reroute at startup.
- **Volume persistence is custom because virtual sinks are recreated each launch.** `VirtualSinks::create()` runs `pw-cli create-node adapter` for the 4 virtual output sinks + 1 mic source on every startup, which resets their volumes to default. We work around this by saving slider changes to `~/.config/arctis-chatmix/volumes.txt` (`save_volume_entry` in `persistence.rs`) and re-applying them in `init_pipeline` immediately after `VirtualSinks::create()` and before `AudioRouter::create`. Only Music / Aux / Mic are persisted: Game and Chat are owned by the ChatMix HID dial (saving them would create a fight between the saved value and the dial position at startup), and Master maps to the physical headset sink whose volume WirePlumber already persists.

### Spatial audio pipeline

- **`set-param` can address ANY named control in the filter graph, not just the capture node's own.** The set-param target is the capture node's global ID, but the parameter keys are `"<node_name>:<ControlName>"` so we can update `mix_out_l:Gain 1` or `eq_l_3:Freq` on the same node. This is what makes the live wet/dry slider work without rebuilding (`update_wet_mix` in `router.rs`).
- **Profile change always rebuilds; wet_mix change uses set-param; both-disabled is a no-op.** The `SetSpatial` handler in `eq_thread` decides via a small match: `(false, false) → no rebuild` (just persist new profile/wet for next toggle), `(true, true)` with same profile → set-param wet update only, anything else → full rebuild. This avoids the ~300 ms audio gap when the user changes profile while spatial is disabled (gap with no audible payoff would be confusing).
- **EQ band updates need the right node prefix when spatial is on.** `update_filter_params` (prefix `eq`) targets the simple chain; `update_filter_params_spatial` (prefix `eq_l` and `eq_r`) targets both per-side chains in one set-param call. Routing decided by `active_spatial_ir(sink, state).is_some()` — keep these branches in sync if you ever rename the EQ chain nodes.
- **HeSuVi WAVs are 14-channel IEEE-float.** Channel order is `[L-leftEar, L-rightEar, LS-leftEar, LS-rightEar, LB-leftEar, LB-rightEar, C-leftEar, C-rightEar, R-leftEar, R-rightEar, RS-leftEar, RS-rightEar, RB-leftEar, RB-rightEar]`. PipeWire's `convolver` filter accepts a `channel = N` config key to extract just one channel from the multi-channel file — we use 14 convolver nodes per sink (one per channel) so we don't have to pre-split the WAVs. Each convolver is mono-in / mono-out.
- **Stereo→7.1 upmix is the canonical EqualizerAPO/HeSuVi fanout** in `build_spatial_graph()`: L→L, R→R, C=0.5(L+R), LS=LB=L, RS=RB=R. Don't try to be clever here — the IR was tuned by HeSuVi's authors with this exact input pattern in mind, so introducing decorrelation/phase tricks at the upmix stage *worsens* the sound (we tried; that's where the "pipey/seashell" complaint originated before this rewrite).
- **Output mixer has 8 inputs (7 wet HRIR + 1 dry bypass).** Wet inputs 1..7 get gain `0.5 * wet_mix`; dry input 8 gets gain `1.0 - wet_mix`. Both input ports are declared in the initial graph config; mixer input counts are fixed at module-load time (you cannot `set-param` your way to more inputs). The dry bypass is what the wet/dry slider crossfades against.
- **Profile dropdown labels are stored separately from persistence keys.** `display_label()` returns the user-facing string ("GSX (SteelSeries GameDAC: Native)"); `key()` returns the stable persistence string ("gsx"). Renaming a label is safe; renaming a key would orphan saved state. Tests assert key uniqueness for that reason.
- **HRIR extraction is per-profile and lazy.** `ensure_ir_file(profile)` is called every time a chain rebuild needs the IR path (idempotent — only writes on size mismatch). Unused profiles never get extracted from the binary blob. Spatial UI greys out the toggle if `hrtf_available()` returns false (any profile failed to extract — usually means the cache dir is unwritable).

### HID

- **HID writer must be a separate file handle, not cloned from the listener's fd**. `try_clone()` on an `O_NONBLOCK` file inherits the flag and writes can fail with `EWOULDBLOCK`. `HidDevice::try_clone_writer()` opens a fresh write-only handle to the same path.
- **udev rule at `udev/71-arctis-nova-elite.rules`** grants `MODE="0666"` on hidraw nodes for VID `1038` PID `2244`. Must be installed to `/etc/udev/rules.d/` on the host.

### App lifecycle

- **Window close = hide, not quit** (Phase 4 behavior). `close_request` handler calls `set_visible(false)` and returns `Propagation::Stop`. The process keeps running with sinks + HID listener alive. Quit is explicit via the "Quit" button at the bottom of the main page, or the tray icon's right-click Quit.
- **Single-instance reactivation**: `connect_activate` fires again when the user re-runs the command while an instance is already running. The handler tracks `setup_done: Rc<Cell<bool>>` so it only builds the window/wires timers once. Subsequent activations just call `present()` on `app.active_window()`.
- **`--hidden` flag only applies to the first activate** — stored in `first_activate_hidden: Rc<Cell<bool>>` which flips to `false` after first use, so later invocations always present the window.
- **`app.run_with_args(&[])` is mandatory** — `app.run()` passes `std::env::args()` to GTK, which rejects `--hidden` as "Unknown option" and crashes the app before it can log anything. We parse `--hidden` ourselves in `main.rs` and pass an empty slice to GTK.

### Persistence

- **Monotonic save rule** (2026-05-14): `save_assignments` and `update_saved_assignment` only ever **add/update** entries that map an app onto a managed sink — they never destroy entries when streams transiently move to unmanaged sinks. Combined with the 3-second `SUPPRESSION_WINDOW` in `reconcile_pure_with_updates`, this means a pactl move announcement that arrives a beat late (eventual consistency) cannot wipe the user's saved mapping. Old `save_assignments` rule (delete-on-leave) was the root cause of Issue 1 in the routing-volume-clips spec; do not revert it without rereading that spec.
- **State-sync only acts on transitions**, not on every observed stream. `state_sync::tick` calls `persistence::reconcile_stream_state` which uses the `tracked: HashMap<u32, String>` map to detect changes since the last tick. Unseen IDs are routed per saved assignments; already-tracked IDs whose sink changed but match `self_move_pre_sink` within 3s are treated as our own moves (not user moves); the rest are treated as user moves and the saved assignment is updated via `update_saved_assignment`. User-initiated moves on existing streams are never undone.
- **Legacy sink name migration** — `sinks.rs` contains `LEGACY_SINK_MIGRATIONS` mapping old `ChatMix_*` names to current `SteelSeries_*` names. `persistence::read_saved()` translates on load; `cleanup_orphaned()` and `cleanup_orphaned_loopbacks()` both destroy legacy-named objects on startup.
- **Preset name is sticky across edits.** `SinkEq.preset_name` used to be cleared when the user dragged bands; it no longer is. The dropdown keeps showing the active preset name even after modifications. Reset (in the Edit popup) depends on this to know what to revert to — `load_built_in(name)` for built-ins, `load_custom_preset(name)` for imported/custom. Consequently, the "Custom" placeholder entry was removed from the dropdown (it existed to represent `preset_name = None`, which no longer happens through normal UI flow). Legacy `eq_state.txt` entries with `"custom"` still load correctly (mapped to `None`) but new saves never write it.

### GTK4 DropDown width

- **`gtk::DropDown` button labels don't ellipsize by default.** The default `SignalListItemFactory` creates plain `gtk::Label`s that request their full natural width, which propagates up through the button to any parent container. `width_request` on an ancestor is a minimum, not a max, so a dropdown with a long item (e.g., "SteelSeries Arctis Nova Elite DAC") will blow out the column width. Fix: assign a custom `SignalListItemFactory` via `dropdown.set_factory(...)` that creates labels with `max_width_chars(1)` + `EllipsizeMode::End`. For the popup list use `set_list_factory(...)` with a separate factory that word-wraps full names. See `truncating_label_factory` / `popup_label_factory` in `src/mixer.rs`.
- **GTK4 CSS does not support per-side borders or `max-width`.** `border-top: 2px solid red` is silently ignored; use `box-shadow: inset 0 2px 0 rgba(...)` for a colored top stripe. Text truncation must be done at the widget level (`set_ellipsize`, `set_max_width_chars`), not via CSS.

### GTK4 symbolic icon SVG rules

**GTK4's fast-path symbolic icon parser ignores `fill` and `stroke` SVG attributes entirely.** It determines rendering solely from the `class` attribute on each element. Without a `class`, every element is **filled solid** (never stroked) — this is why Lucide icons render as dark blobs by default.

**Required for every element** (`<path>`, `<rect>`, `<circle>`):
```xml
class="transparent-fill foreground-stroke" stroke-width="2"
```

**Supported elements**: Only `<path>`, `<circle>`, `<rect>` work. `<line>`, `<polyline>`, `<polygon>`, `<ellipse>` are **silently ignored** — convert `<line x1="A" y1="B" x2="C" y2="D">` to `<path d="M A B L C D">`.

**`stroke-width` must be per-element** — the fast-path parser does not inherit from the root `<svg>`. Keep root-level `fill="none" stroke="currentColor" stroke-width="2"` for rsvg fallback compatibility.

**Template for adding a new icon:**
```xml
<svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24"
     fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round">
  <path class="transparent-fill foreground-stroke" stroke-width="2" d="..." />
  <rect class="transparent-fill foreground-stroke" stroke-width="2" x="..." y="..." width="..." height="..." rx="..." />
  <circle class="transparent-fill foreground-stroke" stroke-width="2" cx="..." cy="..." r="..." />
</svg>
```

Filenames must end in `-symbolic.svg` and be inside a registered icon theme search path (`hicolor/scalable/actions/`).

## Why certain dependencies are absent

- **No `hidapi` crate** — its `linux-native` feature pulls `libudev-sys` which needs `systemd-devel` (not on Bazzite without `rpm-ostree` layering). Raw `/dev/hidraw` + `libc::poll` is sufficient.
- **No `pipewire-rs`** — lacks `create_object` for creating sinks; shelling out to `pw-cli`/`pactl` is what all comparable projects do (nova-chatmix-linux, arctis-nova-mixer, HeadsetControl).
- **No `tokio`** — GTK has its own main loop. HID reads happen on a dedicated `std::thread`, events cross to the GTK main thread via `std::sync::mpsc` polled from a `glib::timeout_add_local`. The `ksni` tray also runs on its own thread and communicates via `mpsc`.
- **No `serde_json` for `pactl`** — `pactl list sink-inputs` output is parsed by hand (line-oriented, predictable format). `serde_json` *is* now a dep, but only for Sonar's web JSON (`src/eq/sonar_import.rs`); continue hand-parsing `pactl` / `pw-cli` output to avoid coupling process state to a schema.

## Sonar preset compatibility

SteelSeries Sonar is Windows-only and doesn't publish its factory presets as data anywhere public. Users still want those presets — the strategy is three layers:

1. **Import from share URL** — `src/eq/sonar_import.rs`. Any user can click "Import Sonar" in the EQ top bar, paste a Sonar share URL (`https://www.steelseries.com/deeplink/gg/sonar/config/v1/import?url=<base64>`) or a direct CDN URL (`https://community-configs.steelseriescdn.com/v1/<uuid>`), and the preset is fetched, parsed, and saved as a custom preset. Works for any Sonar preset — official or user-made — because everything shared through Sonar goes through the same anonymous CDN.
2. **Seed library from a Windows dump** — `tools/dump-sonar-presets.ps1`. A user with Windows + SteelSeries GG runs this PowerShell script, which reads `C:\ProgramData\SteelSeries\GG\coreProps.json`, calls `/subApps` to discover the dynamic Sonar port, and dumps `GET /configs` to one JSON file per preset under `./presets/`. Users can PR those JSONs to seed the repo or drop them into the Linux config dir and import via `sonar_import::parse_sonar_json`.
3. **Release monitoring** — `.github/workflows/sonar-presets.yml`. Daily cron that scrapes `https://techblog.steelseries.com/` for new `GG-notes-*.html` posts, greps for `Added/Updated new EQ Presets`, and opens an issue tagged `new-presets`. State (seen URLs) is persisted in an `actions/upload-artifact` between runs. Names only — values still require Layer 1 or 2.

**Canonical Sonar preset JSON shape**, as stored in `C:\ProgramData\SteelSeries\GG\apps\sonar\db\database.db` (table `configs`, column `data`) and served by the CDN:

```json
{
  "name": "FPS Footsteps",
  "vad": 1,                     // 1=Game 2=Chat 3=Mic 4=Media 5=Aux
  "parametricEQ": {
    "enabled": true,
    "filter1": {"enabled": true, "frequency": 60.0, "gain": 3.0, "qFactor": 1.0, "type": "peakingEQ"}
    /* ...filter10 */
  }
}
```

`type` is one of `peakingEQ | lowShelving | highShelving | lowPass | highPass | notchFilter`. These map 1:1 to `FilterType::{Peaking, LowShelf, HighShelf, LowPass, HighPass, Notch}`. Sonar bounds (freq 20–20000 Hz, gain ±12 dB, Q 0.5–10) fit inside ours; `sonar_import` clamps defensively in case Sonar widens them.

**`FilterType::Notch`** was added specifically for Sonar parity. It's gain-independent (`uses_gain()` returns false), uses Audio EQ Cookbook coefficients (`b0 = 1, b1 = -2·cos(w0), b2 = 1, a0 = 1 + α, a1 = -2·cos(w0), a2 = 1 - α` with `α = sin(w0)/(2Q)`), and shells out to PipeWire's `bq_notch` filter label in `router.rs`. When adding any future filter type, update `FilterType` in `model.rs`, the biquad coefficients in `biquad.rs`, both match statements in `router.rs` (`effective_label` and `build_filter_chain_cmd`), and `parse_filter_type` in `sonar_import.rs`.

## Spatial audio (HeSuVi HRIR convolution)

Per-sink HRTF spatializer that runs on top of the existing EQ chain. Goal is the "speakers in front of you" externalization effect that gamers know from Windows products like Sonar / Atmos / GSX. Uses **HeSuVi**'s curated HRIR library (open-source, the de-facto free alternative to Dolby Atmos on Windows) via PipeWire's built-in `convolver` filter. Available on Game / Music / Aux only; never on Chat or Mic.

**Three bundled profiles** picked from a dropdown:
| Profile | HeSuVi file | Display label |
|---|---|---|
| `Gsx` (default) | `gsx.wav` (~263 KB) | GSX (SteelSeries GameDAC: Native) |
| `Dh2` | HeSuVi's `dh++.wav` (~394 KB) | DH2 (Dolby Headphone: Immersive) |
| `DtsHpx` | HeSuVi's `dtshx.wav` (~66 KB) | DTS (Competitive FPS: Performance) |

All three are 14-channel 48 kHz IEEE-float WAVs extracted from HeSuVi 2.0.0.1 (the SourceForge installer) and embedded into the binary via `include_bytes!`. They live in `data/hrtf/` checked into git. Attribution + provenance is in `LICENSES/hesuvi.txt`. They get extracted to `~/.cache/arctis-chatmix/hrtf/` lazily, only when a profile is actually used.

**Architecture** (when spatial is on for a sink):

```
capture:FL → eq_l_0..eq_l_9 ─┐
capture:FR → eq_r_0..eq_r_9 ─┤
                              │
                         7 upmix mixers:
                           up_l  = 1·L + 0·R    up_r  = 0·L + 1·R
                           up_c  = 0.5·(L+R)
                           up_ls = up_lb = L copies
                           up_rs = up_rb = R copies
                              │
                         14 convolvers, channel = N of HeSuVi WAV:
                           conv_l_lear (ch 0) , conv_l_rear (ch 1)
                           conv_ls_lear (ch 2), conv_ls_rear (ch 3)
                           conv_lb_lear (ch 4), conv_lb_rear (ch 5)
                           conv_c_lear (ch 6) , conv_c_rear (ch 7)
                           conv_r_lear (ch 8) , conv_r_rear (ch 9)
                           conv_rs_lear (10)  , conv_rs_rear (11)
                           conv_rb_lear (12)  , conv_rb_rear (13)
                              │
                         mix_out_l: sum of 7 *_lear outs at gain 0.5·wet
                                  + dry eq_l_9 at gain (1 − wet)        → playback:FL
                         mix_out_r: same with *_rear + dry eq_r_9        → playback:FR
```

Total ~43 nodes per spatial-on sink. With Game/Music/Aux all enabled simultaneously, expect roughly 10–20% CPU on a modern desktop (FFT-based partitioned convolution).

**UI panel** (in `src/eq/mod.rs::build_spatial_panel`): a horizontal `eq-floating-panel` row with title, profile dropdown, an inline `.spatial-mix-pill`-styled Dry/Wet `gtk::Scale` (180px wide, bold "Dry"/"Wet" labels, fixed 30px min-height to match the dropdown height), and the enable switch. The pill's CSS uses `alpha(currentColor, 0.1)` background to match Adwaita button styling.

**Why this approach** (and why an earlier custom-DSP attempt was abandoned): the first iteration tried to hand-roll a stereo→7.1 upmix matrix (Hafler sum/difference decorrelation, allpass-ish nudges) and feed it into PipeWire's SOFA spatializer. The result sounded "robotic / pipey / seashell" because summing N HRTF outputs of correlated content creates audible comb filtering. HeSuVi's IRs were hand-tuned by humans who knew how their 7 channels would be summed back together, so they bake decorrelation into the impulse response itself. We just provide the canonical EqualizerAPO upmix as input and trust the IR to do the rest. The previous SOFA-based code is fully removed; some legacy serialization noise (`pi=…;d=…;fa=…`) is still tolerated in `spatial::parse` for backward-compat with state files written during that prototyping period.

**Game compatibility nuance** (worth saying when users ask why one game sounds great and another sounds muddy): HeSuVi works best on games that output plain stereo with positional encoding baked into pan/phase (most modern FPS, ARC Raiders, Counter-Strike). For games with their own internal HRTF (Valorant, Apex Legends, CS2 with HRTF on, Siege), running HeSuVi on top is double-HRTF and sounds smeared — turn the in-game HRTF on and ours off. The general rule: only one HRTF in the chain at a time.

## Config & state locations

- `~/.config/arctis-chatmix/assignments.txt` — app→sink persistence (`app_name\tsink_name\n` lines)
- `~/.config/arctis-chatmix/eq_state.txt` — per-sink EQ + spatial state. **4 tab-separated fields per line**: `sink_name\tpreset_or_custom\tinline_band_data\tspatial_blob`. The 4th field is `enabled=1;profile=gsx;wet=0.700` (semicolon-separated kv). Older 3-field or 2-field lines parse cleanly with `SpatialState::default()` for the missing data.
- `~/.config/arctis-chatmix/eq_presets/*.txt` — user-saved custom EQ presets (10 lines: `freq\tgain\tq\tfilter_type\tenabled`)
- `~/.config/arctis-chatmix/mixer_routing.txt` — per-channel device routing. Sink lines are 2-field: `SteelSeries_Game\talsa_output.usb-...`. **The mic line is 5 tab-separated fields**: `mic\t<node_name>\t<product_name>\t<bus_path>\t<bluez_address>` (e.g. `mic\talsa_input.usb-SteelSeries-...\tArctis Nova Elite\tusb-0000:00:14.0-3\t\t`). Backward-compatible with legacy 2-field and 3-field mic records — missing fields parse as `None`. `load_mixer_routing` skips the mic line by design; the mic record is read via `load_mixer_routing_mic` into a `MicPreference`. The 4 tiers (exact node → bluez address → bus path → product name) let the hotplug logic re-find the user's preferred device after a name change.
- `~/.config/arctis-chatmix/volumes.txt` — saved mixer slider values (`pipewire_node_name\tvolume_percent\n` lines). Only Music / Aux / Mic are written; Game/Chat/Master are skipped (HID-driven or OS-managed). Captured every 2s by `state_sync::capture_virtual_volumes` and once again on shutdown; restored at `init_pipeline` after `VirtualSinks::create()` and before `AudioRouter::create`.
- `~/.config/arctis-chatmix/clips_settings.txt` — clip-recorder settings. NEW field `save_hotkey_display` (default `"ALT+S"`) — the human-readable label written after `bind_shortcuts` from `proxy.list_shortcuts(&session).await`'s `trigger_description`. Drives the "Hotkey: ALT+S" label on the home-page Clips card.
- `~/.config/autostart/com.github.arctis_chatmix.ArctisNovaEliteChatMix.desktop` — autostart (toggled via the Start at Login switch)
- `~/.cache/arctis-chatmix/icons/hicolor/scalable/actions/*.svg` — extracted Lucide SVGs (recreated on every app launch if size differs)
- `~/.cache/arctis-chatmix/hrtf/*.wav` — extracted HeSuVi HRIR files (`gsx.wav`, `dh2.wav`, `dts-hpx.wav`). Lazily extracted per profile when first needed; size-mismatch triggers re-extraction on next launch.
